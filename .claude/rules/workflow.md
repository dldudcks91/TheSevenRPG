# 구현 워크플로우 (필수 준수)

구현 작업 시 반드시 아래 4단계를 순서대로 진행한다.
단계를 건너뛰거나 사용자 확인 없이 다음 단계로 넘어가지 않는다.

---

## 1단계: 구현 계획 수립

구현 전, 다음 항목을 명시한 계획을 텍스트로 제시한다:
- **무엇을 만드는가** — 기능/목적 한 줄 요약
- **어떤 파일을 변경하는가** — 신규/수정/삭제 파일 목록
- **구현 순서** — 단계별 작업 순서
- **의존성** — 선행 조건이나 연관 모듈

계획 제시 후 사용자 승인을 받는다.

---

## 2단계: 계획 검증

승인된 계획에 대해 아래 항목을 검토한다. 해당 없는 항목은 "해당 없음"으로 명시한다.

### 아키텍처
- 기존 코드와 충돌 여부
- 단일 게이트웨이/Manager 패턴 위반 여부
- 인증 누락 (PUBLIC_API_CODES 해당 여부 확인)

### 어뷰징 방지
- 클라이언트에서 받은 값을 그대로 신뢰하는 곳은 없는가? (서버에서 재검증 필요)
- 동시 요청으로 아이템/골드/경험치 중복 지급 가능한가? (트랜잭션 또는 락 필요)
- 방치형 타이머 등 시간 기반 값을 클라이언트가 조작할 수 있는가?

### 동시성
- Race Condition 발생 가능한 구간이 있는가? (같은 슬롯 동시 장착 등)
- DB 트랜잭션 범위가 적절한가? (원자적으로 처리해야 할 작업 묶음)

### 캐시 일관성
- 이 작업이 Redis 캐시 무효화를 유발하는가? (`battle_stats`, `idle_farm` 등)
- DB 변경 후 Redis 업데이트 누락 가능성은 없는가?

### 수치/상태 경계값
- HP·스탯·골드가 0 이하 또는 음수가 될 수 있는가?
- 상태 전이 순서가 강제되어야 하는가? (예: 전투 시작 없이 전투 완료 호출 차단)

### 클라이언트 효율
- 한 번의 API 호출로 충분한가? (불필요한 N회 왕복 설계 방지)

검증 결과를 간략히 제시하고, 수정이 필요하면 계획을 업데이트한다.

---

## 3단계: 구현

검증된 계획대로 코드를 작성한다.
- 파일 수정 전 반드시 Read로 현재 내용 확인
- 계획에 없던 변경은 하지 않는다
- 구현 중 문제 발견 시 즉시 사용자에게 보고하고 방향 확인

---

## 4단계: 구현 테스트

구현 완료 후:
- 서버 기동 가능 여부 확인 (`python main.py` 또는 `uvicorn`)
- 구현한 API 엔드포인트 동작 확인 (curl 또는 HTTP 요청)
- 에러 케이스 처리 확인
- 테스트 결과를 사용자에게 보고

테스트 실패 시 원인 파악 후 3단계(구현)로 돌아간다.
