# 구현 워크플로우 (필수 준수)

구현 및 리팩토링 작업 시 반드시 아래 6단계를 순서대로 진행한다.
단계를 건너뛰거나 사용자 확인 없이 다음 단계로 넘어가지 않는다.

**참조 스킬:**
- `fastapi-server` — 아키텍처 패턴 (게이트웨이, 데이터 3계층, Redis 패턴)
- `manager-convention` — Manager 코드 레벨 규칙 (메서드 구조, 트랜잭션, 검증, 에러 처리)

**IR 문서 위치:** `fastapi/docs/ir/` (기능별 1파일, 템플릿: `_template.md`)

---

## 0단계: IR 문서 확인 / 작성

구현 시작 전, 해당 기능의 IR 문서를 처리한다.

### 신규 기능인 경우
- `_template.md`를 복사해 `{manager명}.md` 생성
- 사용자와 논의한 동작 규칙 / 입력·출력 / 구현 제약을 채운다
- status를 `planned`로 설정
- **사용자 확인 후** 다음 단계로 진행

### 기존 기능 수정인 경우
- 해당 IR 문서를 Read로 열어 내용을 확인한다
- 변경되는 규칙이 있으면 해당 항목을 먼저 업데이트한다
- 문서가 없으면 신규 기능과 동일하게 소급 작성한다

---

## 1단계: 개발 계획 생성

IR 문서를 기반으로 다음 항목을 명시한 계획을 텍스트로 제시한다:

- **목적** — 기능/목적 한 줄 요약 (IR 문서의 목적 항목 참조)
- **작업 유형** — 신규 개발 / 리팩토링 / 버그 수정
- **변경 파일** — 신규/수정/삭제 파일 목록
- **구현 순서** — 단계별 작업 순서
- **의존성** — 선행 조건이나 연관 모듈

계획 제시 후 **사용자 승인**을 받는다.

---

## 2단계: 개발 계획 검증

승인된 계획에 대해 아래 **6대 관점**으로 검토한다.
각 관점은 `manager-convention` 스킬의 해당 섹션을 판정 기준으로 사용한다.
해당 없는 항목은 "해당 없음"으로 명시한다.

### 2-1. 아키텍처 정합성
> 기존 구조/패턴과 충돌하지 않는가?

- 단일 게이트웨이 / Manager 패턴 위반 여부
- 인증 누락 (PUBLIC_API_CODES 해당 여부 확인)
- 메서드 내부 6단계 구조를 따르는가 (입력추출→메타검증→DB검증→로직→커밋→응답)
- 기존 코드와의 충돌 여부
- **판정 기준**: `manager-convention` §1(메서드 내부 구조), §8(헬퍼 메서드)

### 2-2. 트랜잭션 안전성
> 동시 요청에도 데이터가 안전한가?

- Race Condition 발생 가능한 구간 (같은 슬롯 동시 장착, 골드 동시 차감 등)
- DB 트랜잭션 범위가 적절한가 (원자적으로 처리해야 할 작업 묶음)
- 하나의 메서드에서 commit이 1회인가
- DB 변경 후 Redis 캐시 무효화 누락 가능성
- 캐시 무효화 타이밍이 커밋 이후인가
- **판정 기준**: `manager-convention` §2(DB 트랜잭션), §5(캐시 무효화)

### 2-3. 보안 / 어뷰징 방지
> 클라이언트가 악용할 수 있는가?

- 클라이언트에서 받은 값을 그대로 신뢰하는 곳은 없는가 (서버에서 재검증 필요)
- 소유권 검증이 빠짐없는가 (user_no 조건 필수)
- 동시 요청으로 자원(골드/경험치/아이템) 중복 획득 가능한가
- 시간 기반 값을 클라이언트가 조작할 수 있는가
- 상태 전이 순서가 강제되는가 (전투 시작 없이 전투 완료 호출 차단 등)
- **판정 기준**: `manager-convention` §3(검증 순서), §10(금지 패턴)

### 2-4. 성능
> 불필요한 부하가 있는가?

- API 1건당 DB 쿼리 수 (3개 이하 목표)
- N+1 쿼리 발생 여부
- 한 번의 API 호출로 충분한가 (불필요한 N회 왕복 방지)
- 무거운 연산은 `asyncio.to_thread()`로 분리했는가
- Redis 장애 시 DB 폴백 로직이 있는가
- 동기/비동기 혼용으로 이벤트 루프 블로킹 가능성
- **판정 기준**: `fastapi-server` 성능 체크리스트

### 2-5. 에러 처리
> 실패 시 안전하게 복구되는가?

- 비즈니스 에러(조건문)와 시스템 에러(except) 분리 여부
- 모든 에러가 `error_response(ErrorCode.XXX)` 형식인가
- except 블록에 `logger.error` + `exc_info=True` 포함 여부
- HP·스탯·골드가 0 이하 또는 음수가 될 수 있는가
- 롤백 누락 가능성
- **판정 기준**: `manager-convention` §4(에러 처리), §7(응답 구성)

### 2-6. 유지보수성
> 나중에 수정하기 쉬운가?

- 매직 넘버 없이 상수화 되었는가
- 로깅에 `[ManagerName]` 접두사가 있는가
- 헬퍼 메서드 분리가 적절한가 (너무 긴 메서드 방지)
- 헬퍼가 자체 DB 세션을 생성하지 않고 전달받는가
- **판정 기준**: `manager-convention` §6(로깅), §8(헬퍼), §9(상수)

### 검증 결과 제시 형식

```
| 관점 | 판정 | 비고 |
|------|------|------|
| 아키텍처 정합성 | ✅ 통과 / ⚠️ 수정필요 | 내용 |
| 트랜잭션 안전성 | ✅ / ⚠️ | 내용 |
| 보안/어뷰징    | ✅ / ⚠️ | 내용 |
| 성능           | ✅ / ⚠️ | 내용 |
| 에러 처리      | ✅ / ⚠️ | 내용 |
| 유지보수성     | ✅ / ⚠️ | 내용 |
```

⚠️가 있으면 계획을 수정하고 사용자 재승인을 받는다.

---

## 3단계: 구현

검증을 통과한 계획대로 코드를 작성한다.

- 파일 수정 전 반드시 Read로 현재 내용 확인
- `manager-convention` 스킬의 코드 패턴을 따른다
- 계획에 없던 변경은 하지 않는다
- 구현 중 문제 발견 시 즉시 사용자에게 보고하고 방향 확인

---

## 4단계: 구현 결과 테스트

구현 완료 후 아래를 순서대로 확인한다:

### 4-1. 기동 테스트
- 서버 정상 기동 확인 (`python main.py` 또는 `uvicorn`)
- import 에러, 문법 에러 없음 확인

### 4-2. 기능 테스트
- 구현한 API 엔드포인트 정상 동작 확인 (curl 또는 HTTP 요청)
- 성공 케이스 + 주요 에러 케이스 최소 각 1건씩 테스트

### 4-3. 검증 항목 재확인
- 2단계에서 ⚠️로 지적된 항목이 코드에서 실제로 해결되었는지 확인

테스트 결과를 사용자에게 보고한다.
테스트 실패 시 원인 파악 후 **3단계(구현)** 로 돌아간다.

---

## 5단계: 결과 저장 및 문서 업데이트

테스트 통과 후 아래를 처리한다:

### 5-1. IR 문서 업데이트
- status를 `implemented`(완전 구현) 또는 `partial`(부분 구현)로 갱신
- 구현 중 변경된 규칙·제약이 있으면 해당 항목 수정
- 변경 이력 테이블에 날짜와 내용 추가

### 5-2. SERVER_DEV_PLAN.md 업데이트
- 구현 완료된 항목의 상태를 갱신 (예: `[ ]` → `[x]`)
- 새로 발견된 TODO나 후속 작업이 있으면 추가

### 5-3. 변경 사항 보고
- 변경된 파일 목록
- 추가/수정/삭제된 API 코드
- 후속 작업이 필요한 경우 명시

사용자가 커밋을 요청하면 git commit을 진행한다.
(사용자 요청 없이 자동 커밋하지 않는다.)
